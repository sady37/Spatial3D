## 1. MUSIC 可行性分析

### 用户提问
"如果只计算 100×100×50 的体块，且有高精背景时呢"

### 结论

**全 2D MUSIC 在 C66x 上不可行：**
- 50 检测点 × (特征分解 + 8100 角度搜索) = 100M complex MACs
- C66x @ 900 MMAC/s = **111ms** → 超过 100ms 帧预算
- 用户洞察: "如果 DSP 能实现，汽车座舱雷达早用 MUSIC 了"

**Fall-local MUSIC (仅跌倒附近 5 cells) 可行：**
- 5 cells × focused search (600 angles) = 1.29M MACs = **1.4ms**
- 不需要模式切换 — 天线复数数据每帧已在 DSP 内存中

**Capon (SDK 已有)：**
- ~12° 分辨率 (FFT 29° 的 2-3× 改善)
- 300-600μs / 帧 → 轻松在帧预算内
- 但不是真正的超分辨

## 2. 凝视模式 (Gaze Mode) 设计

### 用户提案
"修改 CFG，调节 Range FOV, AoA FOV 凝视跌倒的地方，只取该 cell 及附近"

### 原理
通过 CLI UART 动态修改 `cfarFovCfg` + `aoaFovCfg` 收窄检测 FOV。不需要改固件。

### 凝视 CFG (ObjectC @ Range≈4.5m, Az≈30°, El≈20°):
```cfg
cfarFovCfg 0 3.5 5.5        # range 仅 2m 窗口
aoaFovCfg 15 45 5 35        # 30°×30° 角度窗口
clutterRemoval 0
cfarProcCfg 0 2 8 4 3 0 2.0 0
cfarProcCfg 1 2 4 2 2 1 2.0 0
```

### 凝视模式实验结果
扫描 ObjectC 区域 3 分钟，对比全房间扫描：
- **凝视模式不提升点密度** — 它只是输出过滤器，不改变物理分辨率
- ObjectC 区域每分钟约 580 个点，凝视和全扫一样
- 价值在于：减少数据量便于上传，不是提升分辨率

### 事件流
```
ESP32 控制 CLI UART:
  ① Doppler 检测下坠 → (R₀, θ₀, φ₀)
  ② sensorStop → 窄 FOV CFG → sensorStart (~200ms)
  ③ DATA UART 录制 10s → 80KB 点云
  ④ WiFi 上传 Server
  ⑤ sensorStop → 恢复全房间 CFG → sensorStart
```

## 3. 3D 能量立方体基线减法

### 方案
```
Zero-doppler 切片: 128 range × 64 az × 16 el = 131K cells
基线存储: 131K × 4B = 524 KB (SRAM 的 23%)
每帧计算: AoA FFT + 减法 + 阈值 = ~0.5ms (帧预算的 0.5%)
```

### 关键洞察
角度模糊 (PSF) 在基线和当前帧完全一致 → 减法后抵消。即使 29° 分辨率，能量变化仍可检测。

### 高精基线存储在 Flash
- MUSIC 分辨率 map = 83KB (64×18×18 cells × 4B)
- 轻松放入 QSPI Flash
- Server 标定时构建，下载到芯片

## 4. 相位数据不可用

### 问题
"难点是，怎么取出指定 cell 的原始相位"

### 结论
OOB 固件所有 TLV 输出 (点云、range profile、heatmaps) 都是**幅度信号**。相位在 AoA FFT 中被消耗。要获取逐天线复数数据，需要最少 40 行固件修改 (新 TLV type，在 AoA FFT 之前拷贝 16 个复数值)。

## 5. 分区扫描 (Zone Scanning)

### 用户提案
"如果这样，我们可以采用相同方式，把整个空间，分区扫描"

### 方案
- 3 个区 × 30° = 90° 总覆盖
- ~165ms/区 (65ms 切换 + 100ms 采集) → 2fps 全房间
- 每区 3× 点密度
- 产品级: Area Scanner (TIDEP-01010) 在 chirp 级做波束转向，不需要 stop/start

## 6. 架构决策: DSP vs Cloud

### 用户方案
"发现 Fall, 划定范围, fall 时，录制该 cell 的 10 秒数据，上传 server 处理。尽量把高级计算放到云端"

### 双层架构
| 层 | 任务 | 延迟 | 算法 |
|----|------|------|------|
| DSP (芯片) | 即时安全判断 | <100ms | FFT AoA + zone 比较 |
| Server (云端) | 精细分类 + 智能分析 | 秒级 | DBN/ML + MUSIC + 语义建图 |

### MUSIC 在 DSP 上的角色
不需要模式切换。每帧天线复数数据已在 DSP 内存。Fall-local MUSIC (5 cells, focused search) 只需 1.4ms，在正常处理管线中直接执行。

但后续讨论推翻了这个方案 — 见第 9 节。

## 8. UART 问题与解决

### 问题
3 分钟凝视扫描后，传感器 DATA UART 无响应。配置发送成功但无数据返回。连续 3 次失败，均报 "stream closed mid-frame"。

### 解决
- 按 EVM 板上 S2 硬件复位按钮
- 手动通过 Python serial 发送配置
- 不带 `--cli-port` 参数收集 (传感器已从手动配置运行)

### 后台任务文件丢失问题
后台收集报告成功但 `gaze_lying_3min.npz` 找不到。原因：后台任务运行时没有 `cd` 到正确目录，相对路径保存到了未知位置。修复：使用绝对路径。

## 9. 基线减法的根本局限 (关键转折)

### 用户洞察
> "椅子也是可以移动的，基线也是可以动的。所以不能靠减法，必须尽可能画现实时的"

### 问题
- 基线减法假设房间不变 → 家具移动后基线失效
- 椅子挪了 → 误报 (或漏检)
- 基线频繁更新 → 可能吸收跌倒者

### 架构转向
```
之前: 当前 - 存储基线 = 变化 → 简单但脆弱
之后: 实时 3D 图 → 直接识别 "人在地面" → 不依赖历史数据
```

这把问题从 **变化检测** 升级为 **场景理解**。

## 10. 稳定系数 + Zone 架构

### 用户方案
> "不能单靠减法。把固定不变的或缓慢变化的，如墙、床、桌子等给一个稳定系数，划出热点区域，按 zone 与基线比较。这些都是要放在 server 上处理后下发。"

### Server 构建语义地图
```
Zone A (墙)     stability = 0.99
Zone B (床)     stability = 0.95
Zone C (桌子)   stability = 0.90
Zone D (椅子区) stability = 0.60  ← 经常移动
Zone E (走道)   stability = 0.30  ← 人来人去
```

### Server 下发到芯片
- `zone_map[]`: 每个 zone 的空间边界
- `baseline[zone]`: 每个 zone 的基线能量
- `threshold[zone]`: 告警阈值 = f(stability)
- `fall_rule`: Z≤0.4m + 面积>0.5m² + 持续>60s

### 椅子移了怎么办
stability=0.6 的 zone 阈值本来就高，椅子位移不触发告警。Server 定期 (每天/每周) 从上传数据重建语义图。

## 11. DSP 资源问题

### 用户质疑
> "每帧 MUSIC → 高清 3D 图 → 直接识别物体 → 更强，DSP 存储及算力够不够，还要考虑其它应用呀"

### DSP 帧预算分析
```
C66x 帧预算 100ms:
  OOB 已用:  CFAR + FFT AoA           ~3ms
  每帧 MUSIC (50 cells):               ~18ms (18%)
  其它应用 (手势/体征/存在):            ~???
```

MUSIC 算法本身需要 200 行 C 固件代码，调试风险高。

### 推荐: 按需 MUSIC
```
正常:    FFT AoA (~3ms)        → zone 能量比较
异常时:  MUSIC 仅异常 zone (~2ms) → 精细确认
```

## 12. 最终架构: 芯片只传数据，Server 做一切

### 用户决定
> "网络带宽是无限的，为什么要在 DSP 上死磕，只要能传走，在 Server 上有大把资源和时间，而且架构更灵活，随时修改"

> "Fall 并不缺少资源。"

### 核心原则
- **芯片是个带 WiFi 的传感器，不是计算平台**
- **所有智能放 Server，随时迭代**
- Fall detection 用 29° 点云 + Server 已验证，不需要 MUSIC
- MUSIC 的意义是画更真实的 3D 图 — 这是独立的产品特性，不是 fall detection 的需求

### 数据量
```
OOB 点云: ~50 点/帧 × 16 字节 × 10 fps = 8 KB/s
UART 1.25 Mbps 使用率: 0.5%
每天: ~691 MB
每月: ~20 GB → 对现代网络/存储不值一提
```

### 最终架构
```
芯片 (零固件改动):
  OOB 固件 → 点云 TLV → UART 1.25Mbps → ESP32 → WiFi → Server
  唯一任务: 传数据

Server (所有智能):
  ├── 实时 3D 建图 (累积点云 → 体素)
  ├── 语义分析 (识别墙/床/桌/椅)
  ├── 稳定系数计算 + zone 划分
  ├── Fall detection (zone 比较 + 时间持续性)
  ├── 历史基线管理 (自适应更新)
  └── ML/DBN 分类 (随时升级模型)
```

### 对比
```
                    DSP 方案          Server 方案
─────────────────────────────────────────────────
固件改动             200 行            0 行
算法更新             重新刷固件        改 Python 重启
调试                 JTAG + CCS        print + log
算力限制             900 MMAC/s        无限
存储限制             256KB SRAM        无限
新算法试验           数天              数分钟
多传感器融合          不可能            天然支持
历史数据回放          不可能            随时
```

### Phase 路线
```
Phase 1 (现在): 29° 点云 → Server 全处理
  够做: fall detection, 存在检测, 活动监测

Phase 2 (如果产品需要精细识别):
  改固件: 输出 per-antenna complex data
  带宽: 16 antenna × 50 det × 32B = 25 KB/s → UART 仍然够
  Server 端跑 MUSIC/Capon/ML beamforming
  芯片还是不做任何计算
```
