硬约束

  • 6s trigger_cancel:只认 anchor R≤0.5m 内 upright+TI静默反证;track lost默认通过
  • 18s 3001_filter:明确STAND/WALK才veto;无数据不否决
  • 首次cube≈Trigger+24s;后续最多2发,查询发起间隔60s
  • 固件 cubeGuard 硬窗口 300s(3000帧)
  • 预算 300 cube-帧/窗口(30s)= 10% 占空
  • 单发上限 300帧(30s);server 用 60帧/发

判决原则

  • Fall ≥ 1:任一发 lying=Y 即报
  • B段presence优先级:RR lock(strength≥0.3) → cube_ff(≥0.5) → z40兜底
  • lying 单独定 fall;Living_state 只贴标签("?"≠崩溃)
  • cube 是权威:进 cube-query = down 已不可信 → down 不再排/撤 cube
  • 红状态机:升红=1发Y · 撤红=连2发N · 作废不算 · Y清零 · 配额只数有效发(Y/N),作废不烧
  • LEG1/2只在red后:LEG1 cloud_up;LEG2 起点0.8m+位移1.5m+限速/瞬移排除
  • R=0.5是Fall身份门;R=0.8是LEG2起点门;10bin是cube回包位置门
  • cube 校验:(A)位置 10bin +(B)归属 query-epoch(只认本次查询回包,发新查询即作废旧包),失败作废 → 下次 query 重查

阈值 已定案

  • cube_ff 强/弱 = 0.5(躺 0.55–0.92 vs 远/静 0.00)
  • z40 lying = 0.4(躺 ~28 ≫ 空 ~0)
  • 每次跌倒 ≤ 3 有效发:相对Trigger约+24/+84/+144s;60s是发起间隔,不是单发采集时长
  • 校验门限 = 10 bin(1bin≈10.8cm → ~1m)
